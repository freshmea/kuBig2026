# 교육자료 검토 보고서

작성일: 2026-04-03

## 요약

현재 저장소는 C 언어, 자료구조, Atmega128, Pico, 네트워크까지 실습 자산이 넓게 준비되어 있다. 강점은 예제 수가 많고 실제 수업 일지가 남아 있다는 점이다. 반면 학습자 입장에서는 "지금 왜 이 예제를 하는지", "다음 프로젝트와 어떻게 이어지는지", "어디까지 해내면 통과인지"가 문서만으로는 선명하지 않다.

이번 검토에서는 다음 세 가지를 핵심 개선 대상으로 본다.

1. 수업 목표와 평가 기준을 예제 단위까지 연결할 것
2. 환경 구축 실패와 빌드 실패를 빠르게 복구할 수 있는 안내를 보강할 것
3. C 기초 문법에서 IoT 프로젝트로 넘어가는 연결 실습을 명시적으로 추가할 것

## 현재 자료의 강점

- `README.md`가 과정 전체 범위를 빠르게 설명한다.
- `doc/c_programming.md`, `doc/atmega128.md`에 실제 수업 진행 이력이 남아 있어 운영 추적이 가능하다.
- `c_example/`, `ds/`, `atmega128/`, `network/`, `pico/`에 실습 자산이 실제 코드 형태로 존재한다.
- `tools/review_course_assets.sh`로 문서 링크와 일부 빌드 자산을 점검할 수 있다.

## 부족한 점

### 1. 성과 기준이 예제와 직접 연결되지 않는다

수업 일지는 상세하지만, 각 차시의 핵심 산출물과 평가 기준이 분리되어 있지 않다. 예를 들어 `scoreProcess`, `baseballGame`, `uart3` 같은 예제가 왜 중요한지, 학습자가 어떤 수준까지 수정할 수 있어야 하는지 문서에서 바로 확인하기 어렵다.

영향:

- 복습 기준이 모호해진다.
- 과제 설계가 강사 개인 기억에 의존한다.
- 신규 강사나 보조 인력이 수업을 이어받기 어렵다.

### 2. 환경 문제 대응 문서가 부족하다

수업 기록에는 WSL, usbipd, udev 설정이 남아 있지만 자주 실패하는 항목이 한 곳에 정리되어 있지 않다. 실제 운영에서는 환경 이슈가 학습 집중도를 가장 크게 깎는다.

영향:

- 강의 시간이 환경 복구에 소모된다.
- 수강생별 진행 속도 차이가 커진다.
- 실습 실패 원인이 코드 문제인지 환경 문제인지 구분하기 어렵다.

### 3. C 기초에서 프로젝트로 넘어가는 다리가 약하다

현재 예제는 문법 설명에는 충분하지만, 최종 프로젝트 문맥과 직접 연결되는 "중간 난이도 과제"가 상대적으로 적다. 특히 파일 입출력, 구조체, 동적 할당을 실제 IoT 데이터 처리 문제로 바꾸는 연결 예제가 있으면 학습 전이가 더 빨라진다.

영향:

- 학생이 "문법 예제는 풀 수 있는데 프로젝트를 못 시작하는" 구간에 머문다.
- 팀 프로젝트 시작 시 요구사항 해석과 데이터 모델링 난도가 갑자기 올라간다.

### 4. 자동 검증 범위가 제한적이다

검토 스크립트는 유용하지만 현재 런타임에 `curl`, `cmake`, `gcc`가 없으면 경고만 남고 실제 품질 확인은 대부분 건너뛴다. 저장소 차원에서 최소 검증 기준을 더 명확히 정의할 필요가 있다.

영향:

- 실습 자료가 실제로 빌드 가능한지 사전에 확신하기 어렵다.
- 운영자가 다른 환경에서 문제를 뒤늦게 발견할 수 있다.

## 즉시 권장하는 개선

### 1. 차시별 "할 수 있어야 하는 것" 문장 추가

`doc/c_programming.md`와 `doc/atmega128.md`의 각 날짜 블록 아래에 한 줄씩 성과 기준을 추가한다.

예시:

- "학생은 `scoreProcess` 입력 파일 크기를 자동 계산하도록 수정할 수 있어야 한다."
- "학생은 UART 수신 버퍼 길이 문제를 재현하고 원인을 설명할 수 있어야 한다."

### 2. 운영 문서 보강

다음 문서를 추가하는 것이 좋다.

- 환경 구축 체크리스트
- 자주 발생하는 빌드/장치 연결 오류 문서
- 과제 제출 템플릿
- 프로젝트 평가 루브릭

### 3. 프로젝트 연결형 예제 추가

이번 작업에서 `c_example/part6/iotLogReport/` 예제를 추가했다. 이 예제는 다음 요소를 한 번에 연습한다.

- 구조체 배열 설계
- 파일 입력 처리
- 동적 메모리 할당
- 포인터 테이블 정렬
- 결과 리포트 출력

문법 복습을 넘어서 "센서 로그를 읽어 상태를 요약하는 프로그램"을 만들게 해, 이후 MQTT/임베디드 프로젝트와 자연스럽게 연결되도록 했다.

### 4. 검증 기준 명확화

가능하면 다음을 표준으로 삼는 것이 좋다.

- 문서 링크 검사
- 호스트 C 예제 `gcc -fsyntax-only`
- 대표 CMake 타깃 빌드
- Atmega/Pico는 도구 미설치 시 스킵하되 설치 방법 링크 제공

## 다음 수업 운영 제안

다음 수업은 새로 추가한 `iotLogReport` 예제를 기준으로 진행하는 편이 적절하다.

진행 순서:

1. 센서 로그 파일 형식 설명
2. 구조체 설계 리뷰
3. 파일에서 데이터 개수 세기
4. 메모리 동적 할당
5. 경고 조건 계산
6. 포인터 배열 정렬
7. 요약 리포트 출력
8. 확장 과제로 MQTT 메시지 형식과 연결

확장 과제:

- 평균 온도 대신 최고/최저 온도도 같이 출력
- 장치별 경고 횟수 정렬
- CSV 형식으로 다시 저장
- 임계값을 사용자 입력으로 받기
