# GitHub 제출 예시

---

이 문서는 팀 프로젝트 저장소를 어떤 형태로 정리하면 평가자와 다른 팀원이 빠르게 이해할 수 있는지 예시를 제공한다.

## 권장 저장소 구조

---

```text
project-name/
├─ README.md
├─ CMakeLists.txt
├─ src/
├─ include/
├─ docs/
│  ├─ proposal.md
│  ├─ presentation.pdf
│  └─ demo.png
└─ assets/
```

## `README.md` 예시 항목

---

1. 프로젝트 한 줄 소개
2. 팀원과 역할
3. 사용한 장치와 기술
4. 빌드 방법
5. 실행 방법
6. 데모 결과 또는 이미지
7. 어려웠던 점과 해결 방법

## 좋은 제출의 특징

---

- 저장소 첫 화면에서 프로젝트 목적이 바로 보인다.
- 실행 명령과 장치 준비 조건이 빠져 있지 않다.
- 발표 자료와 저장소 설명이 서로 다르지 않다.
- 누가 어떤 부분을 맡았는지 적혀 있다.

## 피해야 할 제출 형태

---

- 코드만 있고 실행 방법이 없는 경우
- 발표 자료는 있는데 저장소 README가 비어 있는 경우
- 빌드는 되지만 어떤 기능이 핵심인지 설명이 없는 경우
- 팀원 역할과 기여도가 전혀 드러나지 않는 경우
