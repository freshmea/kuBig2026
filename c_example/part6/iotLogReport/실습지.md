# IoT 로그 분석 실습지

이 실습지는 `iotLogReport` 예제를 이용해 파일 입력, 구조체 배열, 동적 메모리 할당, 포인터 정렬을 한 번에 복습하도록 설계되어 있다.

## 목표

- 센서 로그 파일 형식을 읽을 수 있다.
- 레코드 수를 세고 동적 메모리를 할당하는 이유를 설명할 수 있다.
- 위험 순서 정렬 결과를 보고 우선 대응 장치를 찾을 수 있다.

## 준비

- `sensor_log.dat` 파일 내용을 확인한다.
- `README.md`의 실행 방법을 따라 예제를 한 번 실행한다.
- 생성된 `report.out` 파일을 열어 본다.

## 실습 순서

1. `count_logs`가 어떤 방식으로 데이터 개수를 세는지 설명한다.
2. `load_logs`가 입력 형식이 잘못되었을 때 왜 중간에 멈추는지 확인한다.
3. `analyze_logs`의 경고 조건 3개를 정리한다.
4. `build_sorted_table`이 원본 배열 대신 포인터 배열을 쓰는 이유를 설명한다.
5. `sensor_log.dat`에 새 장치를 2개 추가하고 결과가 어떻게 바뀌는지 본다.

## 체크 질문

- 배열 전체를 정렬하지 않고 포인터 배열만 정렬하면 어떤 장점이 있는가
- 파일 데이터 수를 먼저 세지 않으면 어떤 문제가 생길 수 있는가
- `report.out`에서 가장 먼저 확인해야 하는 값은 무엇인가

## 제출물

- 수정한 `sensor_log.dat`
- 변경 후 `report.out`
- 경고 조건을 본인이 이해한 방식으로 정리한 3문장 요약

## 확장 과제

- 최고 온도 장치와 최저 신호 장치를 함께 출력한다.
- 결과를 CSV 형식으로 저장한다.
- 경고 기준을 사용자 입력으로 받는다.
