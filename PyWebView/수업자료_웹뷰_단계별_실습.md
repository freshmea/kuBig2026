# PyWebView 단계별 수업 자료

## 1. 수업 목표

- WebView 기반 데스크톱 앱의 기본 동작을 이해한다.
- 단순 동작 코드에서 시작해 문제를 발견하고 개선하는 개발 흐름을 경험한다.
- 프론트엔드와 백엔드를 분리한 확장 가능한 구조를 구현한다.
- localhost JSON API를 통해 DOM 텍스트만 갱신하는 방식의 장점을 이해한다.

## 2. 대상 및 사전 준비

- 대상: Python 기초 문법과 HTML/CSS/JavaScript 기초를 배운 학습자
- 준비:
  - Python 3.9+
  - uv
  - 프로젝트 폴더: PyWebView

## 3. 최종 결과물(현재 코드 기준)

- 백엔드: `app/backend/server.py`
  - `/api/clock` JSON API 제공
- 프론트엔드: `app/frontend/index.html`, `app/frontend/style.css`, `app/frontend/app.js`
  - 1초마다 API 호출 후 시간/날짜 텍스트만 갱신
- 앱 부팅: `app/main.py`
  - 로컬 서버 시작 + PyWebView 창 생성
- 실행 진입점: `hello.py`

---

## 4. 수업 진행 시나리오(반복형)

아래는 같은 기능(탁상 시계)을 점진적으로 발전시키는 시나리오다.
각 단계는 반드시 다음 순서를 따른다.

1. 단순 데모 구현
2. 문제점 지적
3. 해결 방법 적용
4. 다음 단계로 확장

## 5. 단계별 실습

### 단계 A. 가장 단순한 데모

#### A-1. 구현

Python에서 PyWebView 창을 띄우고 HTML 문자열을 직접 넣는다.

예시 아이디어:
- `create_window(..., html="<h1>Hello</h1>")`

#### A-2. 문제점 지적

- HTML/CSS/JS가 Python 문자열 내부에 섞여 가독성이 떨어진다.
- 프론트 코드 재사용/확장이 어렵다.
- 팀 협업 시 프론트와 백엔드 역할 분리가 불가능하다.

#### A-3. 해결

- HTML, CSS를 별도 파일로 분리한다.
- 이때 문서 로딩 기준 경로(base context)가 왜 중요한지 설명한다.

강의 포인트:
- "동작한다"와 "유지보수 가능하다"는 다르다.

---

### 단계 B. 파일 분리 후 갱신 방식 문제

#### B-1. 구현

- HTML/CSS 파일을 분리하고 시간을 주기적으로 갱신하려고 시도한다.
- 쉬운 접근으로 문서 전체를 반복 로드하는 방식을 사용해 본다.

#### B-2. 문제점 지적

- 문서 전체 재로딩은 렌더링 비용이 크다.
- 깜빡임/검은 화면 같은 UX 문제가 생길 수 있다.
- UI 상태가 자주 초기화된다.

#### B-3. 해결

- 전체 문서 리로드를 버린다.
- "데이터만 갱신"하는 구조로 전환한다.

강의 포인트:
- 화면 갱신 최소 단위는 "페이지 전체"가 아니라 "필요한 DOM 노드"여야 한다.

---

### 단계 C. 프론트/백엔드 분리 + JSON API

#### C-1. 구현

- 백엔드에서 localhost API를 제공한다.
  - `GET /api/clock`
  - 응답 예시:

```json
{
  "time": "14:23:51",
  "date": "2026년 5월 19일 화요일"
}
```

- 프론트는 1초마다 API를 호출해 텍스트만 바꾼다.

#### C-2. 문제점 지적

- 포트 충돌 가능성
- API 실패 시 화면이 멈출 수 있음
- 예외 처리를 안 하면 학습자가 원인 파악이 어려움

#### C-3. 해결

- 서버 포트를 0으로 바인딩해 사용 가능한 포트를 자동 할당한다.
- fetch 실패를 잡아서 UI 전체가 죽지 않게 한다.
- 로그/에러 처리 규칙을 정한다.

강의 포인트:
- "로컬호스트 통신"은 네트워크 분산 시스템이 아니라, 같은 프로세스 내 역할 분리를 위한 설계 도구로 볼 수 있다.

---

### 단계 D. 최종 구조 완성(현재 프로젝트)

#### D-1. 구조 설명

```text
PyWebView/
  app/
    backend/
      server.py
    frontend/
      index.html
      style.css
      app.js
    main.py
  hello.py
```

#### D-2. 각 파일 역할

- `app/backend/server.py`
  - 정적 파일 서빙
  - `/api/clock` 응답 생성
  - 서버 start/stop 라이프사이클
- `app/frontend/index.html`
  - UI 뼈대
- `app/frontend/style.css`
  - 시계 화면 스타일
- `app/frontend/app.js`
  - `fetch('/api/clock')`
  - `time`, `date` DOM 텍스트 갱신
- `app/main.py`
  - 서버 실행
  - WebView 창 생성
  - 종료 시 서버 정리
- `hello.py`
  - 엔트리포인트

#### D-3. 교육적으로 중요한 결론

- 단순 데모로 시작하면 진입 장벽이 낮다.
- 문제를 직접 경험한 뒤 구조를 개선하면 설계 이유를 이해한다.
- 최종적으로 "분리된 구조 + 작은 API + DOM 부분 갱신"이 확장성과 안정성을 만든다.

## 9. 수업 중 자주 나오는 질문 정리

### Q1. localhost 통신도 네트워크 오버헤드가 큰가?

- 같은 머신 루프백이므로 비용은 작다.
- 대부분의 비용은 문서 전체 리로드/렌더링에서 발생한다.

### Q2. JavaScript를 완전히 없애도 되나?

- 가능은 하지만 실시간 UI 갱신은 오히려 복잡해질 수 있다.
- 교육용 확장성 관점에서는 "작은 JS + 명확한 API"가 더 적합하다.

### Q3. 왜 처음부터 분리하지 않나?

- 학습자는 "문제 체감"이 있어야 설계 이유를 이해한다.
- 그래서 단순 데모 → 문제 노출 → 개선 반복이 더 교육적이다.

## 11. 강사용 운영 메모

- 데모 실패를 두려워하지 말고, 실패를 "문제 정의" 단계로 사용한다.
- 학생에게 정답 구조를 먼저 주지 말고, 문제를 먼저 체험시킨다.
- 최종 구조를 제시할 때는 반드시 파일 역할을 한 줄씩 말하게 한다.
